# Validate Parsed Configuration Data Using Type-Safe Predicates Before Access: Configuration Data Parsed

Status: proposed
Date: 2024-01-09
Deciders: Detection Pipeline (automated)

## Context

- Test suites for MistralVibeAgent and apply-engine parse external configuration files (TOML, JSON) from filesystem paths and environment variables
- Parsed configuration data structures are accessed using .find() predicates to locate specific named entries (mcp_servers, rulerDir paths) without prior schema validation
- Configuration sources include XDG_CONFIG_HOME environment variables and nested directory hierarchies requiring dynamic path resolution
- The codebase uses @iarna/toml for TOML parsing and JSON.parse for JSON deserialization, both of which return untyped 'any' structures

## Problem Statement

Configuration files parsed from external sources (TOML, JSON) produce untyped data structures that are immediately accessed using predicates and property lookups, creating a risk of runtime errors when configuration schemas evolve, files are malformed, or required fields are missing. Without explicit validation between parsing and access, the system cannot distinguish between absent configuration and malformed data, leading to silent failures or type coercion bugs.

## Decision

1. MUST: All configuration data parsed from external sources (TOML, JSON, YAML) MUST be validated against an explicit schema or type guard before property access or predicate-based queries

## Policy Block

- MUST All configuration data parsed from external sources (TOML, JSON, YAML) MUST be validated against an explicit schema or type guard before property access or predicate-based queries

In scope:
- All configuration files loaded from filesystem paths (fs.readFile, fs/promises)
- All configuration data parsed from TOML (@iarna/toml), JSON (JSON.parse), or YAML sources
- Configuration structures accessed using array methods (.find, .filter, .map) or property lookups
- Environment variable-derived configuration paths (process.env.XDG_CONFIG_HOME)

Out of scope:
- Hardcoded configuration constants defined in source code
- Configuration data already validated by external libraries or frameworks
- Test fixtures with known, controlled schemas

Exceptions:
- EXC-001: Configuration parsing occurs within test suites using controlled fixtures with guaranteed schemas

## Rationale

- Evidence shows parsed configuration data typed as 'any' is immediately accessed using .find() predicates without intermediate validation, creating runtime type safety gaps
- The pattern spans multiple configuration sources (TOML, JSON, environment variables) and access patterns (mcp_servers.find, configs.find), indicating a systemic validation gap rather than isolated instances
- Configuration files sourced from XDG_CONFIG_HOME and nested directory hierarchies are subject to user modification and schema drift, requiring defensive validation
- Type-safe validation between parsing and access enables early error detection, clearer error messages, and prevents silent failures from missing or malformed configuration fields

## Consequences

Positive:
- Runtime errors from malformed configuration are caught at validation time with actionable error messages rather than failing deep in business logic
- Type narrowing after validation enables IDE autocomplete and compile-time type checking for configuration property access
- Configuration schema evolution becomes explicit and traceable through schema definition changes rather than implicit property access patterns
- Test suites can verify configuration validation logic independently from business logic that consumes validated configuration

Negative:
- Validation logic adds code complexity and maintenance burden, requiring schema definitions to be kept in sync with configuration file formats
- Performance overhead from validation may be noticeable for large configuration files or frequent reloads, though typically negligible for startup-time configuration
- Overly strict validation schemas may reject valid configuration variations, requiring careful balance between strictness and flexibility

## Alternatives

- Continue using unvalidated 'any' types with defensive null checks at each access site (rejected)
  Rejected because: Defensive null checks do not validate schema structure, only field presence, and scatter validation logic across the codebase rather than centralizing it
  When valid: Only appropriate for throwaway scripts or prototypes with no production usage
- Use TypeScript strict mode with explicit type annotations but no runtime validation (rejected)
  Rejected because: TypeScript types are erased at runtime and cannot validate external configuration files that may not match compile-time assumptions
  When valid: Acceptable only when configuration is generated by type-safe tooling and never manually edited
- Adopt a configuration management library (e.g., convict, config) with built-in validation (deferred)
  Rejected because: Requires larger refactoring to migrate existing configuration loading patterns, though may be appropriate for future architectural evolution
  When valid: When configuration complexity grows to require features like environment-specific overrides, secrets management, or hot reloading

## Risks

- Validation schemas may become stale as configuration file formats evolve, causing false positive validation failures
  Mitigation: Co-locate schema definitions with configuration parsing code and include schema validation in test coverage to detect drift early
  Owner: Engineering team
- Overly strict validation may reject valid configuration variations used in production, causing outages during rollout
  Mitigation: Implement validation in warn-only mode initially, monitor for validation failures in production, then enforce after validation schema is proven stable
  Owner: Engineering team
- Performance overhead from validation may impact startup time for large configuration files
  Mitigation: Profile validation performance during testing, cache validated configuration, and optimize validation logic for hot paths if needed
  Owner: Engineering team

## Implementation Notes

- Start by defining TypeScript interfaces for existing configuration structures (mcp_servers, rulerDir configs) based on observed usage patterns in tests
- Implement runtime type guards or Zod schemas that validate parsed 'any' types against the defined interfaces immediately after parseTOML() or JSON.parse() calls
- Refactor .find() predicates to operate on validated, typed arrays rather than 'any[]' to enable type checking and autocomplete
- Add test cases that verify validation logic rejects malformed configuration and produces clear error messages identifying the schema violation

## Continuation Context


Verify commands:
- grep -r 'parseTOML\|JSON\.parse' --include='*.ts' | grep -v 'validate\|schema\|guard' | wc -l
- grep -r '\.find((.*: any)' --include='*.ts' tests/ src/ | wc -l
- npm test -- --testNamePattern='configuration.*validation' 2>&1 | grep -E 'PASS|FAIL'

Accept when:
- All parseTOML() and JSON.parse() calls in src/ are followed by validation logic before configuration data is accessed
- No .find() or .filter() predicates operate on arrays typed as 'any[]' in production code paths
- Test suite includes validation test cases that verify rejection of malformed configuration with clear error messages

## Enforcement

- Verified by: TypeScript strict mode compilation with noImplicitAny enabled
- Verified by: Code review checklist requiring validation logic for all new configuration parsing code
- Verified by: Automated grep-based verification in CI pipeline detecting unvalidated parse calls
- Violation handling: CI pipeline fails if unvalidated parseTOML() or JSON.parse() calls are detected in new code
- Violation handling: Code review blocks merge if configuration access patterns use 'any' types without validation
- Violation handling: Runtime errors from malformed configuration trigger alerts and rollback procedures
- Exception process: Exception requests must document the configuration source, schema guarantees, and risk assessment
- Exception process: Test code may be exempted if using controlled fixtures with documented schemas (see EXC-001)
- Exception process: Exceptions require approval from tech lead and must include a remediation plan with timeline
