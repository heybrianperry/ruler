<rule_activation id="f3961103-a2e1-47a8-a427-a1d08371e699" title="Standardize Structured Logging with Consistent Format and Levels: Components Implement Log" applies_to="**/*">
These rules are ALWAYS ACTIVE for all logging implementations across the codebase. All new logging code and modifications to existing logging code must comply with these rules.
</rule_activation>

### Rules

- **R-LOG-001** MAY: Components MAY implement log level filtering to control verbosity based on environment or configuration.

### Verify

```bash
# Count direct console usage in src/ (excluding tests)
grep -r 'console\.log\|console\.error' src/ --exclude-dir=node_modules --exclude='*.test.ts' | wc -l

# Count centralized logger imports
grep -r 'import.*logger\|from.*logger' src/ | wc -l

# Run ESLint no-console rule
npm run lint -- --rule 'no-console: error'
```

**Accept when:**
- Direct console.log/console.error usage in src/ is minimal or zero (excluding test files)
- Centralized logger import is present in all modules that perform logging
- Linting rules enforce logging standards and pass in CI pipeline

<enforcement>
Claude Code MUST verify logging compliance using the provided bash commands. Verification is mandatory before accepting logging implementations. ESLint rules configured to flag direct console usage (no-console rule) MUST pass. Code review checklist MUST include verification of proper logging usage. Automated CI checks MUST scan for sensitive data patterns in log statements.
</enforcement>
