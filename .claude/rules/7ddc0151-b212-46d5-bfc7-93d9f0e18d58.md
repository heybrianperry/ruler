<rule_activation id="7ddc0151-b212-46d5-bfc7-93d9f0e18d58" title="Standardize Logging Patterns for CLI and Core Utilities: Error Conditions Logged" applies_to="**/*">
These rules are ALWAYS ACTIVE for all logging implementations in CLI handlers, core utilities, and shared constants modules.
</rule_activation>

### Rules

- **R-LOG-001** SHOULD: Error conditions SHOULD be logged with appropriate severity levels (error, warn) and include stack traces or error details when available.

### Verify

```bash
# Check for existing logging patterns in critical modules
grep -r "console\.log\|console\.error\|console\.warn" src/cli/handlers.ts src/core/FileSystemUtils.ts | wc -l

# Verify logger imports are present
grep -r "import.*logger\|import.*log" src/cli/handlers.ts src/constants.ts src/core/FileSystemUtils.ts

# Find all modules with logging implementations
find src -name '*.ts' -exec grep -l 'logging\|logger\|log\.' {} \; | grep -E '(handlers|constants|FileSystemUtils)'
```

**Accept when:**
- All CLI handlers in src/cli/handlers.ts contain logging statements for command execution, errors, and key operations
- Core utility modules like FileSystemUtils log critical operations (file I/O, errors) with appropriate context
- Logging configuration constants are centralized and referenced by handler and utility modules
- No sensitive information (credentials, tokens, PII) appears in log output during test execution
- Error conditions include appropriate severity levels (error, warn) with stack traces or error details

<enforcement>
Claude Code MUST NOT skip or defer verification of logging patterns in CLI handlers, core utilities, and shared constants modules. All error conditions MUST be logged with appropriate severity levels and contextual information.
</enforcement>
