<rule_activation id="be4d3575-40b8-4b6b-b282-34888f9ca9dd" title="Standardize Structured Logging with Consistent Format and Levels: Components Use Structured" applies_to="**/*">
These rules are ALWAYS ACTIVE for all logging operations within the system. All components that emit log messages must comply with the structured logging standards defined herein.
</rule_activation>

### Rules

- **R-LOG-001** MUST: All components MUST use a structured logging framework with consistent format across the application.
- **R-LOG-002** MUST: All production code including CLI handlers, core utilities, and business logic MUST use structured logging.
- **R-LOG-003** MUST: Third-party library integrations that emit application logs MUST use the standardized structured logging framework.
- **R-LOG-004** MUST: Error handling and exception logging across all layers MUST use structured logging.
- **R-LOG-005** MUST: Operational events such as startup, shutdown, and configuration changes MUST use structured logging.
- **R-LOG-006** MUST: All log entries MUST include required metadata fields (timestamp, level, component, operation, correlation_id) in consistent format.
- **R-LOG-007** SHOULD: Use appropriate log levels: ERROR for failures requiring attention, WARN for degraded states, INFO for significant operational events, DEBUG for detailed troubleshooting.
- **R-LOG-008** SHOULD: Implement centralized log aggregation to collect logs from all components into a searchable platform.
- **R-LOG-009** MAY: Performance-critical hot paths may request exceptions where logging overhead is measured and documented as unacceptable (EXC-001).
- **R-LOG-010** MAY: Legacy components undergoing gradual migration may request exceptions to new logging standards (EXC-002).

### Verify

```bash
# Check for unstructured logging statements
grep -r 'console\.log\|print(' src/ | wc -l  # Should return 0 or minimal count for unstructured logging

# Verify structured logging usage
grep -r 'logger\.(error|warn|info|debug)' src/ | wc -l  # Should show structured logging usage

# Run logging-specific tests
npm test -- --grep 'logging'  # Run logging-specific tests to verify format compliance
```

**Accept when:**
- All production code uses the standardized logging framework with no unstructured console.log/print statements
- Log entries include required metadata fields (timestamp, level, component) in consistent format
- Automated tests verify log output format and sensitive data redaction
- Code review checklist includes verification of appropriate log levels and structured format
- No unstructured logging detected in production code paths via automated linting
- Log format compliance verified through automated tests in CI pipeline
- Static analysis tools confirm no potential sensitive data logging
