<rule_activation id="1daff665-5ed6-49bb-9ae6-aabea6d59523" title="Standardize Test File Organization for CI/CD Pipeline Validation: Test Files Organized" applies_to="tests/**/*.test.ts">
These rules are ALWAYS ACTIVE for all test files within the CI/CD pipeline and apply to all agent-related test suites, configuration validation tests, and integration tests.
</rule_activation>

### Rules

- **R-TEST-001** MUST: Test files MUST be organized with clear separation between unit tests (tests/unit/), integration tests (tests/integration/), and configuration tests (tests/)

### Verify

```bash
# Count test files matching the expected organization pattern
find tests/ -name '*.test.ts' | grep -E '(unit/agents/|integration/|claude-|gemini-|mcp-)' | wc -l

# Verify agent test files exist in unit directory
grep -r 'describe.*Agent' tests/unit/agents/ | wc -l

# Verify security-critical input validation tests exist
grep -r 'input.*validation\|type.*field\|MCP.*config' tests/ | wc -l
```

**Accept when:**
- All agent implementations have corresponding test files in tests/unit/agents/ directory
- Configuration validation tests exist for all security-critical input fields (type, HTTP config, MCP settings)
- Test organization follows the three-tier structure: unit tests, integration tests, and configuration tests with clear naming conventions
- Test file naming conventions are followed: tests/unit/agents/{AgentName}.test.ts for unit tests, tests/integration/{feature-name}.test.ts for integration tests, and tests/{config-aspect}.test.ts for configuration validation

<enforcement>
Claude Code MUST NOT skip or defer verification. CI/CD pipeline blocks merge if required tests are missing or failing. Code review process flags missing test coverage for new agents or configuration changes.
</enforcement>
