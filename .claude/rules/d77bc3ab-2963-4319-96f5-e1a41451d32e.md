<rule_activation id="d77bc3ab-2963-4319-96f5-e1a41451d32e" title="Standardize Configuration Loading with Unified Environment Management: Agent Implementations Not" applies_to="**/*">
These rules are ALWAYS ACTIVE for all agent implementations, configuration loaders, and runtime environment management components. All code that loads configuration, manages environment variables, or initializes runtime settings MUST comply with these rules.
</rule_activation>

### Rules

- **R-CONFIG-001** MUST: Agent implementations MUST NOT directly access process.env or raw configuration files; they MUST use the configuration abstraction layer (UnifiedConfigLoader or approved equivalent).
- **R-CONFIG-002** MUST: All configuration values loaded through the abstraction layer MUST be validated and sanitized before use to prevent injection attacks and runtime errors.
- **R-CONFIG-003** MUST: Configuration schemas MUST be defined with type-safe interfaces (TypeScript interfaces or Zod schemas) to enable compile-time type checking.
- **R-CONFIG-004** MUST: Configuration source precedence MUST be enforced as: CLI args > environment variables > config files > defaults.
- **R-CONFIG-005** SHOULD: Configuration loading SHOULD include comprehensive debug-level logging with sensitive values redacted.
- **R-CONFIG-006** SHOULD: Configuration access patterns SHOULD implement caching for validated configuration values to minimize performance overhead.

### Verify

```bash
# Count direct process.env accesses outside UnifiedConfigLoader
grep -r 'process\.env\[' src/ --exclude-dir=node_modules | grep -v UnifiedConfigLoader | wc -l

# Run configuration validation tests
npm run test:config -- --grep 'configuration validation'

# Check for ESLint violations on direct process.env access
eslint src/ --rule 'no-restricted-syntax: [error, {selector: "MemberExpression[object.object.name=process][object.property.name=env]", message: "Use UnifiedConfigLoader instead of direct process.env access"}]'
```

**Accept when:**
- All agent implementations (OpenCodeAgent, MistralVibeAgent, ZedAgent, CodexCliAgent, AmazonQCliAgent, GeminiCliAgent, FirebenderAgent, RooCodeAgent) load configuration exclusively through UnifiedConfigLoader or approved abstraction layer
- Configuration validation tests pass for all supported configuration sources (TOML, environment variables, VSCode settings, MCP)
- No direct process.env accesses exist outside of the UnifiedConfigLoader implementation itself
- All configuration schemas are defined with type-safe interfaces and validation logic
- Configuration loading includes proper error handling and fallback mechanisms
- Sensitive configuration values are redacted in logs and error messages

<enforcement>
Claude Code MUST NOT skip or defer verification of configuration loading patterns. All violations of R-CONFIG-001 through R-CONFIG-006 MUST be identified and remediated before code acceptance. ESLint rules preventing direct process.env access MUST pass in CI pipeline.
</enforcement>
