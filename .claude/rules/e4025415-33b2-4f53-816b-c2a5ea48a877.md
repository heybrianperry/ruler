<rule_activation id="e4025415-33b2-4f53-816b-c2a5ea48a877" title="Standardize Logging Patterns for CLI and Core Utilities: Core Utility Modules" applies_to="**/*">
These rules are ALWAYS ACTIVE for all logging implementations in CLI handlers, core utilities, and shared constants modules.
</rule_activation>

### Rules

- **R-LOG-001** MUST: Core utility modules (such as FileSystemUtils) MUST include logging for critical operations, errors, and state changes.

### Verify

```bash
# Check for logging implementations in key files
grep -r "console\.log\|logger\|log\." src/cli/handlers.ts src/constants.ts src/core/FileSystemUtils.ts

# Check for logging imports
grep -r "import.*log" src/cli/handlers.ts src/constants.ts src/core/FileSystemUtils.ts

# Find all files with logging patterns
find src -name '*.ts' -exec grep -l 'logging\|logger\|log\.' {} \;
```

**Accept when:**
- All three key files (src/cli/handlers.ts, src/constants.ts, src/core/FileSystemUtils.ts) contain consistent logging implementations
- Logging constants are centralized and imported by modules that need them
- New CLI handlers and core utilities include appropriate logging for errors and critical operations

### Scope

**In scope:**
- CLI command handlers in src/cli/handlers.ts
- Shared logging constants in src/constants.ts
- Core utility modules such as src/core/FileSystemUtils.ts
- All public API contracts that expose operational behavior
- Error handling and exception logging across all layers

**Out of scope:**
- Third-party library logging (unless wrapped by application code)
- Test fixtures and mock logging implementations
- Development-only debug statements not intended for production
- External service logging (handled by those services)

### Exceptions

- **EX-001:** Performance-critical code paths where logging overhead is measured and documented as unacceptable
- **EX-002:** Security-sensitive operations where logging could expose sensitive data

**Exception process:**
- Document the exception request with rationale (performance, security, or technical constraints)
- Submit for review by architecture review board or designated technical lead
- If approved, document the exception in code comments and architecture decision log
- Periodically review exceptions to determine if they can be resolved

<enforcement>
Claude Code MUST NOT skip or defer verification of logging pattern compliance. Code review checklist MUST include verification of logging pattern compliance. Automated grep-based checks in CI pipeline MUST detect logging patterns. Periodic architecture reviews MUST assess logging consistency across modules.
</enforcement>
