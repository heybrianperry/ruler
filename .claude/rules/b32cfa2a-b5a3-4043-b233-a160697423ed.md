These rules are ALWAYS ACTIVE for all runtime environment configuration and cache layer implementations. All configuration management code MUST follow these patterns.

### Rules

- **R-CACHE-001** SHOULD: Configuration changes SHOULD propagate reactively without requiring component restarts where technically feasible.

### Verify

```bash
# Verify cache configuration pattern usage in propagate and settings files
grep -r 'cache.*config' src/ --include='*.ts' | grep -E '(propagate|settings)' || echo 'Pattern not found'

# Count settings and propagate files containing cache references
find src/ -name '*settings*.ts' -o -name '*propagate*.ts' | xargs grep -l 'cache' | wc -l

# Run cache configuration tests
npm test -- --grep 'cache.*configuration' 2>/dev/null || echo 'No cache config tests found'
```

**Accept when:**
- Cache configuration is declared in centralized modules and referenced by at least 2 runtime components
- Configuration propagation code exists in MCP and VSCode extension contexts
- Tests verify that cache settings changes propagate correctly across component boundaries

<enforcement>
Claude Code MUST NOT skip or defer verification of cache layer configuration management patterns. All pull requests introducing cache-related functionality MUST be verified against these rules before acceptance.
</enforcement>
